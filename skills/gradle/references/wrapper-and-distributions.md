# Gradle Wrapper And Distributions

Read this when: wrapper files, wrapper generation or upgrades, distribution selection, checksum, download retry, installation state, private distributions, or wrapper scripts own the work.

## Scope Boundary

- This file owns `gradlew`, `gradlew.bat`, `gradle-wrapper.jar`, `gradle-wrapper.properties`, the `wrapper` task, and distribution download and installation behavior.
- The checked-in scripts and Wrapper JAR are caller-side components; `distributionUrl` selects the downloaded Gradle distribution but does not upgrade those checked-in files.
- Read [runtime-and-structure.md](runtime-and-structure.md) for client/daemon boundaries, Daemon JVM policy, Gradle user home resolution, VFS/file watching, or init scripts.
- Read [ci-and-security.md](ci-and-security.md) for CI wrapper validation, secret injection, cache trust, or action-specific wrapper policy.
- Read [upgrade-strategy.md](upgrade-strategy.md) when the wrapper change is part of a broader Gradle upgrade or deprecation migration.

## Generate And Upgrade

- The wrapper is the Gradle version of record. Prefer wrapper commands over globally installed `gradle`.
- Treat wrapper scripts and `gradle-wrapper.jar` as pre-build executable trust roots; dependency verification, repository policy, and build logic safeguards do not run before them.
- If wrapper files are missing, generate them with a trusted installed Gradle or trusted Gradle distribution by running the `wrapper` task in the project; do not download or copy unaudited wrapper scripts or JARs into the repository.
- Upgrade with the `wrapper` task, not only by editing `distributionUrl`; run it once for properties and again when scripts/JAR should be refreshed.
- Since Gradle 9, `gradle-wrapper.properties` requires `X.Y.Z` versions even when the `wrapper` task can resolve labels or major/minor selectors.
- Use `-bin` distributions by default; choose `-all` only for offline or air-gapped source/docs needs, or when policy explicitly requires the larger distribution.
- When build logic or help text needs the supported wrapper distribution types, read `Wrapper.DistributionType.values()` directly instead of deprecated `Wrapper.getAvailableDistributionTypes()`.
- Gradle 8.2 or newer provides `validateDistributionUrl` and `--no-validate-url` on the `wrapper` task. This is generation-time reachability policy carried by the Gradle distribution running the task, not a runtime integrity control carried by the checked-in Wrapper JAR; disabling it does not make a later download trusted or replace distribution/JAR checksum validation.
- When refreshing Gradle 9.5+ wrapper or application Windows scripts, review custom `.bat` callers/templates for changed environment scoping, `CALL`/exit behavior, and removed exit-environment-variable assumptions.

## Download And Installation

- Use `distributionSha256Sum` from the Gradle release checksums for the distribution ZIP; validate `gradle-wrapper.jar` separately because a ZIP checksum does not prove the checked-in JAR.
- The ZIP checksum is checked before a new installation, while a valid cached installation with its `.ok` marker is reused without re-downloading or re-hashing; prove checksum policy with a clean or controlled Gradle user home.
- If `gradle-wrapper.properties` contains `distributionSha256Sum`, keep wrapper task configuration or CLI checksum input in sync when regenerating wrapper properties.
- `networkTimeout` requires a checked-in `gradle-wrapper.jar` generated by Gradle 7.6 or newer and applies to each download connection and read; an older Wrapper JAR silently ignores it even when `distributionUrl` targets a newer release.
- `retries` and `retryBackOffMs` require a checked-in `gradle-wrapper.jar` generated by Gradle 9.5 or newer; an older Wrapper JAR silently ignores them even when `distributionUrl` targets a newer release. `retries` is the number of extra attempts and `retryBackOffMs` is the initial delay that doubles after each failed attempt; use bounded retries for transient I/O, not to hide authentication, HTTP policy, or checksum failures.
- Wrapper installation writes a `.part` download, locks the target ZIP, renames only a completed download, and writes `.ok` only after the extracted distribution is valid; concurrent first runs should share one installation. An exclusive-access timeout points to another wrapper holding the distribution lock, a stalled wrapper process, or unreliable shared-filesystem locking; cache-directory permission failures surface earlier while creating or opening the lock path. Neither is a task-execution failure.
- Invalid ZIP recovery evaluates at most three completed archive candidates, the initial archive plus up to two redownloaded archives. This archive-validation loop is separate from the configured network retries that may occur while fetching any one candidate. If `is no valid zip file` persists, inspect proxies or mirrors returning HTML/truncated content and verify the checksum before increasing retries.
- Custom distributions must extract to exactly one top-level directory, contain the Gradle launcher JAR, and use safe archive entry paths; failures at this stage are distribution-shape or archive-safety problems, not daemon compatibility problems.
- For private wrapper distributions, use HTTPS plus host-scoped credentials or tokens in user/CI properties. Do not commit shared credentials in `distributionUrl`.

## Wrapper Review

- Treat `./gradlew wrapper --gradle-version <version>` as a mutating upgrade command. Review scripts, JAR, properties, URL validation, retry/timeout policy, checksums, and CI entrypoints together; a Wrapper JAR checksum matching a different Gradle version usually means properties changed without regenerating the JAR.
- Check whether the command uses the checked-in wrapper or a globally installed Gradle.
- Check whether CI and developers use the same wrapper scripts, JAR, properties, and distribution checksum policy.
- Check whether private wrapper credentials are host-scoped and kept in user/CI properties.
